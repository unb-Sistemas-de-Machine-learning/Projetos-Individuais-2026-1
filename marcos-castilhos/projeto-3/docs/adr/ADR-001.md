# ADR-001: Arquitetura de Roteamento Determinístico e Extração Estrita

> **Data:** [05/05/2026]
> **Status:** aceita

---

## Contexto

Por vezes tentei gerenciar os meus gastos financeiros manualmente (olhando no app do banco e registrando na planilha). Essa estratégia, ao longo do tempo, se mostrou ineficaz. Há algum tempo já vinha planejando desenvolver essa solução para me servir de assistente pessoal mas não tinha tempo. Com o trabalho tive a oportunidade de implementar esse projeto.

---

## Alternativas consideradas

### Alternativa A: Abordagem simples baseada em prompt

- **Descrição:** Uso de um LLM gerando texto livre (baseado em mensagem no Telegram) e tentativa de extração via Regex ou código Python posterior.
- **Prós:** Fácil implementação inicial no n8n.
- **Contras:** Altíssima taxa de quebra de contrato. Regex falha frequentemente com variações de resposta do LLM, tornando o sistema frágil.

![fluxo-a](/projeto-3/solutions/solution-a/fluxo-solution-a.png)

### Alternativa B: RAG com Base de Conhecimento

- **Descrição:** Criar embeddings do histórico financeiro para tentar adivinhar valores baseados em compras passadas.
- **Prós:** Pode autocompletar informações faltantes.
- **Contras:** Introduz o risco de inventar valores financeiros irreais. Complexidade acidental desnecessária para o escopo do problema.

![fluxo-b](/projeto-3/solutions/solution-b/fluxo-solution-b.png)

### Alternativa C: Fluxo Multi-etapas com Structured Outputs e Fail-Fast

- **Descrição:** Uso de inferência atômica forçando o schema JSON, combinada com pré-processamento de áudio (spoofing de MIME type para o Groq Whisper) e roteamento estrito no n8n via nó Switch. 
- **Prós:** Comportamento análogo a um compilador (Fail-Fast). Erros são isolados e não afetam o banco de dados.
- **Contras:** Maior tempo de setup na orquestração visual e exigência de injeção de código JavaScript no fluxo.

![fluxo-c](/projeto-3/image.png)

---

## Decisão

_Qual alternativa foi escolhida e por quê?_

A Alternativa C foi aceita. Em sistemas financeiros, a precisão e a rejeição de dados incorretos são mais importantes que a flexibilidade. O uso do Llama com "Structured Outputs" acoplado ao nó If transfere a responsabilidade da validação para o orquestrador, criando um fluxo intrinsecamente auditável.

---

## Consequências

_Quais os impactos dessa decisão? O que muda no projeto?_

O sistema não terá memória de conversação (stateless). Mensagens ambíguas serão ativamente rejeitadas, forçando o usuário a reenviar o dado com a formatação mental correta. Há necessidade de manter o snippet JS de conversão .oga para .ogg funcional.
